# @nodev.builder Instagram Reference System v1

작성일: 2026-05-03

## 목적

앞으로 카드뉴스를 감각으로 만들지 않는다. 레퍼런스 게시글을 계정별·게시글별·슬라이드별로 기록한 뒤, 그중 어떤 규칙을 빌리고 어떤 규칙은 버릴지 결정한 다음 제작한다.

## 핵심 원칙

1. **커버만 보면 안 된다.** 캐러셀은 내부 슬라이드 구조가 핵심이다.
2. **게시글 1개 = 여러 장의 이미지 묶음**으로 기록한다.
3. **레퍼런스 → 템플릿 → 커버 시안 → 확장** 순서로만 진행한다.
4. **6장 전체 제작 전에는 direction board만 만든다.**
5. **얼굴/손/주요 오브젝트 위 텍스트 박스 금지.**
6. **예쁜 이미지보다 이미지-카피 연결성이 먼저다.**

## 레퍼런스 DB 구조

- `reference-schema.json`: 게시글/슬라이드 기록 필드
- `seed-inventory.json`: 4개 계정 1차 게시글 후보 목록
- `accounts/*.md`: 계정별 분석 노트
- `posts/*.json`: 게시글 단위 상세 기록
- `templates/*.json`: @nodev.builder 적용 템플릿
- `direction-boards/*.md`: 제작 전 사용자 확인용 방향 보드

## 멀티슬라이드 기록 방식

캐러셀 게시글은 아래처럼 기록한다.

```text
Post
├─ Cover: 피드에서 클릭하게 만드는 역할
├─ Slide 02: 문제/상황/시각 리듬 전환
├─ Slide 03: 핵심 구조/방법
├─ Slide 04: 증거/예시/프롬프트/결과
├─ Slide 05: 저장 가치/체크리스트/템플릿
└─ Slide 06+: CTA/요약/댓글/프로필 링크
```

각 슬라이드는 다음 항목을 반드시 채운다.

- role: 이 장이 왜 필요한가
- layout: 이미지/텍스트/박스/프레임 배치
- image_role: 이미지는 분위기/증거/결과물/인물/제품 중 무엇인가
- text_role: 제목/설명/CTA/라벨 중 무엇인가
- transition: 이전 장에서 왜 이 장으로 넘어오는가
- reusable_rule: @nodev.builder에 가져올 수 있는 규칙
- avoid_rule: 그대로 가져오면 안 되는 점

## 제작 전 게이트

새 카드뉴스를 만들기 전 아래 질문에 답하지 못하면 제작하지 않는다.

1. 이 게시글은 어떤 레퍼런스 계정의 어떤 게시글 3개를 섞는가?
2. 커버 카피 공식은 무엇인가?
3. 2~6장은 정보형인가, 캠페인형인가, 혼합형인가?
4. 인물 이미지는 몇 장까지 쓸 것인가?
5. 실제 결과물/UI/워크플로우 증거는 어느 장에 들어가는가?
6. CTA는 댓글 키워드인가, 프로필 링크인가, 저장 유도인가?
7. 사용자에게 먼저 보여줄 커버 시안은 몇 개인가?

## 다음 수집 작업

1. 각 계정 10개씩 게시글 URL/스크린샷 확보
2. 캐러셀 내부 슬라이드 캡처
3. 게시글당 `posts/{account}-{번호}.json` 작성
4. 패턴을 묶어 `templates/nodev-template-*.json` 정의
5. 템플릿별 커버 1장만 테스트

## 이번 실패에서 생긴 금지 규칙

- 가독성 해결을 위해 얼굴 위에 큰 패널 얹기 금지
- 모든 장을 image2 강한 이미지로 채우기 금지
- 반대로 이미지 없이 밋밋한 템플릿 카드뉴스로 회귀 금지
- 레퍼런스 없이 “트렌디하게”라는 말만으로 제작 금지
