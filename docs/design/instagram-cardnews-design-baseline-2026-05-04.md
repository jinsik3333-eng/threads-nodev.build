# 비개발dev 인스타 카드뉴스 디자인 기준점 v1

작성일: 2026-05-04
대상: `@nodev.builder` Instagram / Threads 보조 콘텐츠
목적: Hermes/Claude가 즉흥 미감으로 카드뉴스를 만들지 않도록, 레퍼런스·폰트·이미지·레이아웃·검수 기준을 고정한다.

---

## 0. 한 줄 기준

> 비개발자가 “AI로 내 일을 직접 만들 수 있겠다”고 느끼게 만드는, 읽기 쉬운 실무형 에디토리얼 콘텐츠.

디자인은 예쁘기보다 먼저 아래를 만족해야 한다.

1. 첫 1초에 멈춘다.
2. 한글 문장이 자연스럽다.
3. 모바일에서 작은 글씨까지 읽힌다.
4. 이미지가 장식이 아니라 메시지를 돕는다.
5. 카드뉴스 템플릿/PPT/강의 광고처럼 보이지 않는다.

## 1. 콘텐츠 유형은 2개만 쓴다

### A. 설명형 — Warm Minimal / Claude Editorial

사용할 때:
- Claude Code, AI 업무 시스템, MCP, Subagent, Context, 자동화 개념 설명
- 비개발자의 오해·불안·실수 교정
- 저장형 실무 카드뉴스

참고:
- 토픽/후킹: `@gymcoding`
- 시각 톤: Claude Editorial, Warm Minimal, Notion/Sanity식 콘텐츠 우선 레이아웃

핵심 인상:
- 차분함
- 신뢰감
- “아 이건 저장해야겠다”
- 코딩 강의가 아니라 업무 정리 문서 같은 느낌

금지:
- 형광 노랑/파랑 강조
- 전부 볼드
- 과한 아이콘/이모지
- 둥근 정보 박스 남발
- 템플릿식 카드뉴스 느낌

### B. 트렌디형 — Premium Campaign / Human + UI Font Mix

사용할 때:
- 콘텐츠 자동화, 브랜드 워크플로우, AI로 결과를 만드는 장면
- “이게 AI로 된다고?” 같은 욕망/결과 후킹
- 계정 성장용 커버/캐러셀

참고:
- `@mobileeditingclub`
- `@adrianabubori`
- 보조: `@sebia.ai`는 어두운 브리핑/스토리텔링 무드 참고

핵심 인상:
- 고급 브랜드 캠페인
- AI/Claude를 라이프스타일과 결과물로 보여줌
- 큰 타이포 + 실제로 갖고 싶은 결과물
- 사람/손/제품/작업 공간/프롬프트 UI가 의미 있게 섞임

금지:
- 6장 전부 같은 인물 포스터처럼 만들기
- 얼굴/손/주요 오브제를 텍스트 박스로 가리기
- HTML/SaaS 랜딩페이지처럼 보이는 내부 슬라이드
- 이미지와 카피가 따로 노는 것
- 프롬프트 UI 텍스트가 안 보일 정도로 작아지는 것

## 2. 레퍼런스 사용 규칙

디자인 전에 반드시 레퍼런스를 “느낌”이 아니라 구조로 해부한다.

각 레퍼런스마다 아래를 적는다.

```yaml
reference:
  account: "@..."
  post_url: "..."
  visual_family: "warm-editorial | premium-campaign | dark-briefing | dark-tech"
  first_1s_hook: "처음 눈에 들어오는 것"
  image_subject: "인물 / 손 / 제품 / 화면 / 공간 / 결과물 / 없음"
  image_job: "욕망 / 증거 / 대비 / 맥락 / 장식"
  headline_position: "상단 / 중앙 / 하단 / 이미지와 겹침"
  headline_lines: ["의미 단위 줄바꿈"]
  text_hierarchy: "제목 > 보조 > CTA > 라벨"
  layout_pattern: "cover / problem / solution / workflow / proof / CTA"
  borrow: "가져올 점"
  reject: "버릴 점"
  adapt: "우리 계정에 맞게 바꿀 방식"
```

통과 기준:
- `image_job`을 설명하지 못하면 그 이미지는 쓰지 않는다.
- `borrow/reject/adapt`가 없으면 레퍼런스를 봤다고 치지 않는다.
- 실제 이미지가 있는 경우 텍스트 요약만 보고 흉내 내지 않고 vision 분석을 먼저 한다.

## 3. 한글 카피 기준

### 문장 톤

좋은 톤:
- 비개발자 직장인이 이해하는 말
- 번역투가 아닌 자연스러운 한국어
- 과장보다 “아 맞다” 싶은 문제 인식
- 도구명이 아니라 결과/상황 먼저

나쁜 톤:
- `무료 가이드는 열어두고` 같은 어색한 직역체
- `생산성을 극대화하세요` 같은 SaaS 광고문
- `AI 혁명`, `10배 성장` 같은 빈 과장
- 설명형인데 캠페인 카피처럼 너무 뜬 문장
- 트렌디형인데 강의 목차처럼 긴 문장

### 커버 제목 줄바꿈

좋은 예:

```text
코딩 몰라도,
내 일은
직접 만들 수 있습니다
```

```text
콘텐츠 30일치
Claude가 짜줌
```

나쁜 예:

```text
코딩
몰라도 내
일은 직접
만들 수 있습니다
```

규칙:
- 의미 단위 2~3줄 우선.
- 조사/어미만 한 줄에 남기지 않는다.
- 제목은 한 번에 읽혀야 한다.
- 강조는 크기 차보다 색상/위치/여백으로 먼저 한다.

## 4. 타이포그래피 기준

### 공통

- 모바일에서 읽어야 하는 최소 텍스트: 22px 이상.
- 작은 라벨도 장식이면 과감히 삭제한다.
- 모든 글자를 볼드로 만들지 않는다.
- 제목, 본문, 보조, CTA의 굵기 차이를 유지한다.

### 설명형 기본값

```yaml
font_family: Pretendard
headline_weight: 650-720
body_weight: 400-520
helper_weight: 400-500
headline_line_height: 1.08-1.18
body_line_height: 1.45-1.65
letter_spacing: -0.02em ~ -0.01em
```

인상:
- 단정함
- 명확함
- 고급 설명서

### 트렌디형 기본값

```yaml
korean_headline: SUIT 우선
support_text: Pretendard 또는 SUIT
ui_text: IBM Plex Sans KR / IBM Plex Mono / JetBrains Mono 계열
headline_weight: 700-850
body_weight: 450-600
english_accent: italic serif 또는 restrained sans, 필요할 때만
```

인상:
- 캠페인형
- 선명하지만 유치하지 않음
- 한글이 영어 레퍼런스보다 둔해 보이지 않게 비례 조정

금지:
- Gmarket Sans 기본 사용 금지. 프리미엄 캠페인에는 대체로 광고 썸네일처럼 보임.
- Noto Sans KR 기본값 남발 금지. 너무 일반적일 수 있음.
- 제목 900 weight 기본 금지. 다크 테크형이 아니면 둔해진다.

## 5. 컬러 기준

### 설명형 팔레트

```yaml
background: "#F7F2EA / #F6EFE5 / #FAF7F2"
ink: "#1F1B18 / #2B2520"
muted: "#6F6258"
line: "rgba(43,37,32,0.16)"
accent: "#B96A45 / #C46A45 / #8E5A3D"
```

규칙:
- 배경 1개, 잉크 1개, 강조 1개 중심.
- 강조색은 한 장에서 1~2곳만.
- 버튼/박스 색보다 여백과 타이포 위계로 읽히게 한다.

### 트렌디형 팔레트

밝은 캠페인형:

```yaml
cream: "#F4EBDD"
sky: "#AFC7D8"
white: "#FFFFFF"
charcoal: "#171513"
sage: "#87937D"
terracotta: "#B46B4B"
```

어두운 브리핑형:

```yaml
black: "#101010"
ink: "#EDEAE2"
muted: "#A8A096"
cyan_or_blue: "필요 시 소량"
border: "rgba(237,234,226,0.18)"
```

금지:
- 형광 노랑 + 파랑 라벨 조합을 기본값으로 쓰지 않는다.
- 네온/글로우/과한 그라데이션으로 세련됨을 만들려고 하지 않는다.
- 한 카드에 강조색을 3개 이상 넣지 않는다.

## 6. 이미지 기준

### 좋은 이미지의 조건

이미지는 “예쁜 배경”이 아니라 아래 중 하나의 역할을 해야 한다.

- 욕망: 저렇게 일하고 싶다.
- 증거: 실제 결과물/화면/작업물이 보인다.
- 맥락: 비개발자의 업무 상황이 보인다.
- 대비: Before/After 차이가 보인다.
- 스토리: 다음 장을 넘기고 싶게 만든다.

### 트렌디형 이미지 후보 점수표

```yaml
image_candidate:
  hook_score: 0        # 1초 정지력
  subject_score: 0     # 주제와 직접 연결
  text_space_score: 0  # 제목 안전영역
  premium_score: 0     # 고급감
  ai_artifact_score: 0 # AI 티 적음
  contrast_score: 0    # 글자 대비 가능
  total: 0
  verdict: "use / revise / reject"
```

자동 탈락:
- 얼굴/손/키보드/텍스트 왜곡이 눈에 띔.
- 가짜 글자가 선명하게 보임.
- 제목 넣을 공간이 없음.
- 소품이 너무 많아 시선이 흩어짐.
- 예쁘지만 콘텐츠 주제와 연결되지 않음.

### image2 사용 원칙

- image2에게 완성 카드뉴스를 맡기지 않는다.
- image2는 텍스트 없는 재료 이미지, hero visual, product/workspace scene, UI mockup용이다.
- 한글 타이포와 최종 조판은 별도로 한다.
- 이미지 mood가 좋아도 글자가 안 읽히면 editorial panel/overlay로 해결하되, 얼굴/손/핵심 오브제를 가리면 실패다.

## 7. 레이아웃 기준

### 공통 안전영역

```yaml
canvas: "1080x1350"
outer_margin: "72-96px"
header_y: "48-72px"
bottom_safe_area: "120px 이상"
page_number: "top-right only"
account_label: "top-center"
```

규칙:
- 페이지 번호를 중복 배치하지 않는다.
- 상단 계정명 위치를 슬라이드마다 바꾸지 않는다.
- CTA는 하단 너무 끝에 붙이지 않는다.
- 1px라도 겹침이 애매하면 실패다.

### 커버 레이아웃 패턴

A. Magazine Cover
- 여백 40~60%.
- 제목 2~3줄.
- 박스 거의 없음.
- CTA는 작은 캡션/캡슐.

B. Brand Campaign Poster
- 인물/제품/작업 장면이 60~70%.
- 제목이 이미지 일부를 덮을 수 있으나 얼굴/핵심 오브제는 살린다.
- 설명문 최소화.

C. Warm Editorial Declaration
- 문장 하나를 크게.
- 얇은 라인, 작은 캡션, 적은 정보량.
- 첫 게시글/정체성 선언에 적합.

D. Dark Briefing
- 검은 배경 + 명확한 정보 위계.
- `sebia.ai`식 차분한 브리핑은 가능.
- Halfdone식 형광 썸네일은 현재 기본 기준에서 제외.

### 내부 슬라이드 패턴

설명형:

```text
문제 인식
→ 오해 교정
→ 핵심 개념
→ 실무 적용
→ 체크리스트/CTA
```

트렌디형:

```text
Campaign Hero
→ Problem / Time Cost
→ Skill or Workflow Output
→ Problem 반복
→ Output / Artifact
→ CTA
```

내부 슬라이드 금지:
- 모든 장이 같은 포스터.
- 설명 박스만 쌓은 HTML 랜딩페이지.
- 목적 없는 아이콘 네트워크.
- 목업 텍스트가 안 보여서 장식이 되는 UI.

## 8. 작업 프로세스 기준

절대 바로 6장을 만들지 않는다.

```text
1. Brief 작성
2. Reference 3개 이상 vision/구조 분석
3. 스타일 패밀리 선택
4. 카피/스토리보드 작성
5. 이미지 후보 3개 또는 레이아웃 후보 3개
6. 커버 3시안 제작
7. contact sheet + vision QA
8. 사용자 선택
9. 선택된 방향으로 6장 확장
10. 전체 contact sheet QA
11. 위험 카드 개별 PNG 확대 검수
12. 최종 PNG + 캡션 + CTA/DM 문구 전달
```

중요:
- 사용자가 스타일을 싫어하면 같은 스타일을 계속 다듬지 않는다.
- 커버 방향이 승인되기 전에는 2~6장을 만들지 않는다.
- 레퍼런스 문법 없이 HTML/PIL로 흉내 내지 않는다.

## 9. QA 체크리스트

### 카피 QA

- [ ] 첫 장만 봐도 이 계정이 무엇을 말하는지 안다.
- [ ] `1~3편` 같은 내부 용어를 외부 카드에 쓰지 않았다.
- [ ] `입문 3단계`처럼 처음 보는 사람도 이해하는 표현을 썼다.
- [ ] 한국어가 번역투가 아니다.
- [ ] 각 슬라이드의 역할이 다르다.

### 타이포 QA

- [ ] 제목 줄바꿈이 의미 단위다.
- [ ] 모바일에서 작은 글씨도 읽힌다.
- [ ] 모든 요소가 볼드가 아니다.
- [ ] 본문/보조/라벨의 위계가 있다.
- [ ] CTA가 너무 작거나 하단에 붙지 않았다.

### 이미지 QA

- [ ] 이미지 역할이 장식이 아니다.
- [ ] 얼굴/손/핵심 오브제를 가리지 않았다.
- [ ] AI 티가 눈에 띄지 않는다.
- [ ] 가짜 글자가 보이지 않는다.
- [ ] 제목 안전영역이 충분하다.

### 레이아웃 QA

- [ ] 계정명/페이지 번호 위치가 일관된다.
- [ ] 1px 애매한 겹침이 없다.
- [ ] 박스가 제목보다 먼저 보이지 않는다.
- [ ] 여백이 충분하다.
- [ ] contact sheet에서 전체 덱이 한 시스템으로 보인다.

### 최종 산출 QA

- [ ] 모든 PNG는 1080×1350.
- [ ] contact sheet 생성 완료.
- [ ] 위험 카드 개별 확대 검수 완료.
- [ ] 캡션/댓글 키워드/DM 문구가 카드 내용과 연결된다.

## 10. 현재 기준의 결론

지금 비개발dev 인스타 디자인의 우선순위는 다음이다.

1. **정체성**: 코딩 강의가 아니라 비개발자의 AI 작업 시스템.
2. **문장**: 한국어가 자연스럽고 처음 보는 사람도 이해해야 한다.
3. **구조**: 카드마다 narrative job이 있어야 한다.
4. **가독성**: 모바일에서 작은 글씨까지 읽혀야 한다.
5. **레퍼런스**: 실제 이미지/게시물 분석 후 변형해야 한다.
6. **미감**: 박스/색/볼드가 아니라 사진·여백·타이포 위계로 만든다.

이 문서는 다음 작업부터 기본 디자인 헌법으로 사용한다.
